# 5년치 보도자료 — 저장 및 활용 아키텍처 (Govpress MCP / Govpress 확장)

## 0. TL;DR

> **전제 재확인**: Govpress MCP는 독립 서비스가 아니라 **Govpress_md_pdf의 확장**이다. Govpress는 정책브리핑 **API 호출(크롤) + HWPX→MD 변환** 로직을 이미 코드로 보유. 단 `api2.govpress.cloud` 웹서비스는 소량 on-demand용이며 파일을 저장하지 않는다. 따라서 5년 백필 같은 **대량 처리는 Govpress 코드를 서버W 로컬에서 직접 실행**(파이썬 모듈 임포트 또는 CLI)하고, 결과물(HWPX·MD)은 Govpress MCP가 자체 디렉토리에 보관·색인한다. 원본도 PDF가 아니라 **HWPX**다(이전 설계 문서의 "PDF" 언급은 HWPX로 정정).

Codex의 3가지 질문에 대한 결론부터:

| # | 질문 | 권장 답 | 한줄 근거 |
|---|---|---|---|
| 1 | 사용 방식 | **둘 다** (원문회수 + RAG 청크) | 도구 1·2·5·6은 원문회수 계열, 3·4·7·8은 RAG 계열. 한쪽만 쓰면 절반의 도구가 동작 안 함. |
| 2 | 원본 보관 | **MD+메타+원본 모두** (단 분리 저장) | HWPX 원본은 재변환·감사·검증에 필수. 공개 리포에는 MD+메타만, 원본은 서버W 로컬 스토리지에. Govpress 웹서비스는 저장하지 않으므로 Govpress MCP가 자체 디렉토리에 처음부터 보관한다. |
| 3 | 갱신 방식 | **초기 백필 + 일일 증분 + 주간 정합성 스캔** | 증분만 하면 누락 감지 불가, 재스캔만 하면 낭비. 둘을 조합하되 재스캔은 저빈도. |

Codex가 묻지 않았지만 중요한 결정 축:

| # | 추가 결정 | 확정 답 |
|---|---|---|
| 4 | 공개 범위 분리 | 공개 리포(MD+메타) + 비공개 자산(원본 HWPX, statistics, 임베딩, 운영 로그) |
| 5 | 재구축 가능성 | 벡터 DB·검색 인덱스는 모두 "MD+메타로부터 결정론적 재생성 가능"을 불변조건으로 설계 |
| 6 | 중복·동일성 판정 | `briefing_id`(뉴스ID)만으로 unique — 같은 ID에 다른 sha256이면 compare_versions 후보(검증 대기) |
| 7 | API 범위 | 정책브리핑 API가 돌려주는 **전체** (중앙부처 + 지자체). 화이트리스트 없음. `entity_type`으로 계층 구분 |
| 8 | 인프라 | **서버W (WSL Ubuntu 24.04, i9/64GB/RTX 3080, Docker Desktop, Cloudflare Tunnel)** 단일 호스트 |
| 9 | 공개 엔드포인트 | `mcp.govpress.cloud` (Govpress 서브도메인) |
| 10 | 임베딩 | BGE-M3 셀프호스팅, GPU(RTX 3080) 가속 |
| 11 | 크롤·변환 코드 | Govpress가 보유한 **크롤 + HWPX→MD 변환 로직**을 재사용. 대량 처리는 `api2.govpress.cloud` 웹서비스를 호출하지 않고 **같은 코드를 서버W 로컬에서 직접 실행**. 결과물(HWPX·MD)은 Govpress MCP가 자체 보관 |

## 1. 볼륨 추정

바탕 수치가 없으면 설계 사이징이 공중부양한다. 먼저 거칠게라도 추정.

### 1.1 건 수

2021-04-17 ~ 2026-04-17 전체(중앙부처 + 지자체) 보도자료 추정치:

- **중앙부처**: 대부처(18개) 각 연 800~1,500건 + 소부처·위원회 각 연 200~500건 + 대통령실·국무조정실 각 연 300~600건 → 연 10,000~12,000건
- **광역지자체** (17개): 각 연 1,500~3,000건 → 연 25,000~50,000건
- **기초지자체** (226개, API 노출되는 부분만): 각 연 200~800건 추정 → 연 40,000~180,000건

어림잡아 **연 75,000~240,000건 → 5년 380,000~1,200,000건**. 안전 상한을 **150만 건**으로 잡되, Phase A는 **중앙부처 + 광역지자체 ≈ 5년 20만~30만 건**까지만 우선 적재 후 기초지자체는 Phase B로 분리.

> 참고: Govpress_md_pdf는 크롤·변환 **로직을 코드로 보유**하지만 웹서비스(api2.govpress.cloud)는 소량 on-demand용이고 산출물을 저장하지 않는다. Phase A는 Govpress 코드를 **로컬에서 직접 실행**해서 전체 크롤 + 전체 변환 + 자체 저장을 처음부터 수행. 5년 백필은 korea.kr API rate limit에 묶여 2~4주 예상.

### 1.2 파일 크기

실측이 없으니 업계 통상치로(건 수는 Phase A 상한 30만 기준):

| 자산 | 건당 | 30만 건 총량 |
|---|---|---|
| HWPX 원본 | 평균 300KB (100KB~1.5MB) | ~90GB |
| Markdown 변환물 | 평균 15KB | ~4.5GB |
| Frontmatter(YAML) | 0.5KB | ~150MB |
| 청크 임베딩 (BGE-M3, 평균 8 chunks × 1024 float32) | 32KB | ~9.6GB |
| statistics 레코드 | 평균 4건×200B = 0.8KB | ~240MB |
| FTS5 인덱스 | MD 대비 1.5배 | ~7GB |
| Qdrant HNSW 오버헤드 | 임베딩 × 1.3 | ~12GB |

**총합 ≈ 124GB**, 이 중 공개 대상(MD+frontmatter) ≈ 4.7GB.

이 볼륨이 결정을 갈라친다.

- 4.7GB는 단일 GitHub 리포에는 부담스러움 → **연도별/권역별 분할 리포** 또는 **git-annex/DVC** 검토
- 90GB HWPX는 Git 네이티브에 부적합. 서버W 로컬 디스크에 보관(이미 디스크는 충분하다고 함)
- 전체 124GB는 서버W 단일 디스크로 여유. 클라우드 이관 시에만 비용 이슈

## 2. 목표와 제약

**기능 목표**
- 8개 MCP 도구가 P95 지연 SLO 내에 답할 수 있어야 함 (명세 §9.2 기준)
- 단일 개발자가 유지보수 가능 (운영 복잡도 낮을수록 좋음)
- 공개 데이터 자산으로서도 가치 (타인이 fork·재사용 가능)

**비기능 제약**
- 재난 복구: HWPX 원본만 살아있으면 전 시스템 재구축 가능해야 함
- 관찰성: 어떤 보도자료가 언제 적재·업데이트됐는지 추적 가능
- 공개 민감성: 공개 리포에는 실무자가 공유받고 싶지 않을 수 있는 `statistics` 테이블을 두지 않음
- 비용: 서버W가 이미 가동 중이므로 **마진 비용 월 $5 이하** 목표 (Haiku 라벨링 토큰비가 거의 전부)
- Govpress 간섭 금지: 기존 api2.govpress.cloud 서비스의 응답 지연·디스크를 침해하면 안 됨 (별도 스토리지 경로·리소스 제한 필수)

## 3. 3계층 스토리지 아키텍처

구성 요소를 **Hot / Warm / Cold** 3계층으로 분리. 역할이 섞이지 않도록 각 계층의 **source of truth** 를 명확히 정의.

```
┌───────────────────────────────────────────────────────────────┐
│ COLD — 재난 복구용 원본 (서버W 로컬)                          │
│  · HWPX 원본 (WSL 네이티브 경로, /home/USER/govpress-mcp/data/raw/…) │
│  · Govpress MCP가 Govpress 크롤 코드를 bulk 실행해 로컬 저장             │
│  · 일일 크롤 로그 (어떤 URL을 언제 받아왔나)                  │
│  · 변경되지 않음, 쓰기 전용, 읽기 드묾                         │
│  · 장기 백업: rclone으로 주 1회 R2 또는 외장 디스크에 미러    │
└───────────────────────────────────────────────────────────────┘
                            │ 재변환 필요 시
                            ▼
┌───────────────────────────────────────────────────────────────┐
│ WARM — 지식 소스 오브 트루스                                  │
│  · policy-briefing-kr (공개 GitHub 리포, 권역별 분할)         │
│  · 연도/부처/YYYY-MM-DD-slug.md + frontmatter                 │
│  · manifest.json (전체 인덱스)                                │
│  · Git 커밋 이력 = 변경 감사 로그                             │
└───────────────────────────────────────────────────────────────┘
                            │ 결정론적 재생성
                            ▼
┌───────────────────────────────────────────────────────────────┐
│ HOT — 질의 처리용 파생 인덱스 (Docker Compose 스택)           │
│  · SQLite (briefings, statistics, entity_alias, manifest)     │
│  · SQLite FTS5 (전문 검색 가상 테이블)                        │
│  · Qdrant (청크 임베딩, 필터 메타) — Docker 컨테이너          │
│  · BGE-M3 임베딩 서버 (TEI, RTX 3080 GPU 바인딩)              │
│  · Redis (옵션, 핫셋 500건 LRU 캐시)                          │
│  · Govpress MCP 서버 (Python, SSE transport)                      │
└───────────────────────────────────────────────────────────────┘
                            │ Cloudflare Tunnel
                            ▼
                    mcp.govpress.cloud
```

**핵심 불변조건**: Hot 계층은 **언제든 버리고 Warm으로부터 재생성** 가능해야 한다. 즉 Hot에만 있는 데이터가 없어야 한다. 이 규칙이 deployment, 마이그레이션, 장애 대응을 단순하게 만든다.

### 3.1 Cold — 원본 보관 (서버W 로컬)

- **저장소**: 서버W 로컬 디스크. 디스크는 "매우 많음"이라 용량은 부담 없음. Cloudflare R2는 **미래 이관 옵션**으로만 보존(egress 비용·재난 복구 필요 시).
- **경로 규약** (WSL 네이티브 경로 필수 — `/mnt/c/...` Windows 드라이브에 두면 I/O 성능 급락):
  ```
  /home/USER/govpress-mcp/data/
  ├── raw/                       # HWPX 원본 (Govpress MCP 자체 보관)
  │   └── {year}/{entity_type}/{entity}/{briefing_id}.hwpx
  ├── md/                        # 변환 결과 (Git 리포로도 커밋)
  │   └── {year}/{entity}/{YYYY-MM-DD}-slug.md
  ├── fetch-log/                 # 일일 크롤 로그 (JSONL)
  │   └── {YYYY-MM-DD}.jsonl
  └── checksums.db               # sha256 인덱스 (SQLite)
  ```
- **Govpress 연계**: Govpress_md_pdf는 크롤·변환 로직을 코드로 보유하지만 웹서비스는 저장 기능이 없다. Govpress MCP는 **Govpress 레포를 프로젝트에 편입**(git submodule 또는 editable install)해 import로 크롤·변환 함수를 호출하고, 입출력 파일은 모두 Govpress MCP 자체 경로(`data/raw`, `data/md`)에 보관한다. api2.govpress.cloud 웹서비스는 bulk 실행 시 호출하지 않는다.
- **백업 정책**: 주 1회 `rclone sync`로 Cloudflare R2 버킷 또는 외장 디스크에 미러. 서버W 장애 시 복구 RTO < 1일.
- **접근**: 일상 운영에서는 읽기 없음. 재변환 필요 시 또는 감사 시에만. Docker 컨테이너에는 **읽기 전용 마운트**로 노출.
- **대안 평가**:
  - Cloudflare R2: egress 무료 S3 호환, 90GB 저장 월 ~$1.35. 지연·대역폭이 로컬보다 느리지만 감수 가능. **Phase C 이관 후보**.
  - Git LFS: 대역폭 제한(월 1GB) 때문에 기각.
  - Backblaze B2: R2 대비 저장 $0.005/GB 차이 미미, egress 요금 존재해 R2가 낫다.

### 3.2 Warm — 공개 지식 베이스

- **저장소**: `github.com/USER/policy-briefing-kr` (공개, MIT 또는 CC-BY-4.0)
- **구조**:

```
policy-briefing-kr/
├── README.md
├── manifest.json           # 전체 인덱스: briefing_id → path
├── LICENSE
├── 2021/
│   ├── 기획재정부/
│   │   ├── 2021-04-20-2022년예산안편성방향.md
│   │   └── ...
│   └── 산업통상자원부/
├── 2022/
├── 2023/
├── 2024/
├── 2025/
├── 2026/
├── schemas/
│   ├── frontmatter.schema.json
│   └── manifest.schema.json
└── scripts/
    ├── verify.py          # frontmatter 스키마 검증
    └── rebuild_manifest.py
```

- **frontmatter**: 명세 §0.1의 규약 유지. 필수 필드: `id`, `title`, `entity`, `entity_type`(central/metro/local), `date`, `url`, `sha256`, `revision`, `extracted_by`, `raw_path`(서버W 상대경로) + 선택 `raw_r2`(미러 URL)
- **파일명 규약**: `{YYYY-MM-DD}-{slug}.md`, 슬러그는 제목 한글 그대로 하이픈 치환, 50자 캡
- **Git 사용 원칙**: 1보도자료 = 1커밋 원칙 안 강제 (초기 백필은 배치 커밋). 다만 증분 수집 이후 정정 커밋은 1커밋으로 명확히 분리.
- **manifest.json**: 로컬에서 재생성 가능. 포맷:

```json
{
  "generated_at": "2026-04-17T10:00:00Z",
  "total": 52847,
  "entries": {
    "156720115": {
      "path": "2026/과학기술정보통신부/2026-01-15-디지털플랫폼정부2026년추진계획.md",
      "title": "디지털플랫폼정부 2026년 추진계획",
      "ministry": "과학기술정보통신부",
      "date": "2026-01-15",
      "sha256": "a3f5...",
      "revision": 2
    }
  }
}
```

manifest가 있으면 Hot 계층을 재빌드할 때 filesystem walk 없이 바로 iterate.

### 3.3 Hot — 파생 인덱스 (Docker Compose 스택)

Hot 계층 전체를 서버W의 **단일 docker-compose 프로젝트**로 운영. 각 컨테이너 역할:

```yaml
# docker-compose.yml (발췌)
services:
  qdrant:
    image: qdrant/qdrant:v1.11
    volumes: ["./data/qdrant:/qdrant/storage"]
    ports: ["127.0.0.1:6333:6333"]    # 내부 전용

  embed:                                  # BGE-M3 임베딩
    image: ghcr.io/huggingface/text-embeddings-inference:turing-1.5
    command: ["--model-id", "BAAI/bge-m3", "--max-batch-tokens", "8192"]
    deploy:
      resources:
        reservations:
          devices: [{capabilities: ["gpu"], count: 1}]  # RTX 3080 1장
    volumes: ["./models:/data"]
    ports: ["127.0.0.1:8080:80"]

  redis:
    image: redis:7-alpine
    command: ["redis-server", "--maxmemory", "1gb", "--maxmemory-policy", "allkeys-lru"]
    ports: ["127.0.0.1:6379:6379"]

  mcp:                                    # Govpress MCP 서버
    build: ./mcp
    environment:
      QDRANT_URL: http://qdrant:6333
      EMBED_URL: http://embed:80
      SQLITE_PATH: /data/govpress_mcp.db
      RAW_ROOT: /data/raw                 # ro 마운트
    volumes:
      - ./data/sqlite:/data
      - /home/USER/govpress-mcp/data/raw:/data/raw:ro
    ports: ["127.0.0.1:8000:8000"]        # cloudflared가 참조
```

- **SQLite**: 호스트 경로 `./data/sqlite/govpress_mcp.db`. Phase A 상한 30만 건 기준 SQLite 파일 ~1GB 수준, 여전히 SQLite가 편안한 영역.
  - 테이블: `briefings`, `briefing_entities` (공동발표 처리), `entity_alias`, `statistics`, `revision_summaries`
  - FTS5 가상 테이블 `briefings_fts`: 제목·본문 토큰화 인덱스 (한국어 토크나이저는 `unicode61 remove_diacritics 2` + `trigram` fallback, mecab-ko는 별도 실험 후 도입)
- **Qdrant**: 단일 컨테이너. 컬렉션 `briefing_chunks` 하나.
  - 벡터 차원: BGE-M3 기준 1024
  - 페이로드: `briefing_id`, `entity`, `entity_type`, `date`, `section`, `chunk_idx`
  - HNSW 파라미터: `m=16`, `ef_construct=200`, 리콜 우선. 30만 건 × 8 청크 ≈ 240만 벡터 → 인덱스 ~12GB, 쿼리 지연 30만 점에서도 < 50ms.
- **Embed (TEI + BGE-M3)**: GPU 바인딩 필요. Govpress가 RTX 3080을 다른 작업에 쓰고 있다면 **실행 시간대 조율 또는 MIG/프로세스 우선순위 조정** 필요. 초기 백필 때만 장시간 점유, 증분은 하루 수 분 수준.
- **Redis**: `search_briefing` 결과·정책 타임라인 캐시. 트래픽 적으면 생략 가능하지만 docker-compose 상 유지 비용 0에 가까워 기본 포함.
- **MCP 서버**: Python `mcp` SDK, FastAPI + SSE 트랜스포트. 내부 127.0.0.1만 바인딩하고 외부는 Cloudflare Tunnel이 전달.

Hot 계층 리소스 상한 (i9 / 64GB / RTX 3080 서버W):
- Qdrant 메모리: 4~6GB (30만 건 기준)
- TEI 임베딩: GPU VRAM ~3GB, 시스템 RAM 4GB
- SQLite 버퍼: 2GB
- Redis: 1GB cap
- MCP 서버: 1GB
- 여유분: 나머지 ≥ 45GB → Govpress 병행 운영 여유.

### 4.2 원본 보관 — **MD+메타+원본 모두, 단 분리 저장**

"MD+메타만" 옵션의 위험:
1. Govpress 변환 코드가 개선되면 재변환 불가. 5년 후 Govpress v1.0이 나왔을 때 v0.3으로 변환된 자료를 업그레이드 못 함.
2. 감사·법적 이슈 발생 시 "우리가 원본을 본 적 있다"를 증명 불가.
3. 변환 버그로 숫자가 틀렸는데 원본이 없으면 recovery 불가.

"MD+메타+원본 모두"로 하되 공개 리포와 원본 스토리지를 분리하는 이유:
1. GitHub 리포 크기 폭주 방지 (90GB HWPX vs 4.7GB MD는 유지보수 난이도 차원이 다름)
2. 공개 리포는 포크·스타·이슈로 여러 사람이 접근, 원본은 읽기 드묾 — 트래픽 패턴이 다름
3. 저작권 리스크 격리: MD는 추출·재구성물(derivative)이라 "fair use + 메타정보" 논거 가능, HWPX 원본 재배포는 별도 이슈

구체 실행:

```
GitHub public (MD+메타)       ─┐
                               │
서버W 로컬 (HWPX 원본, ro 마운트)┼─→ derive_hot.py가 둘 다 끌어와 Hot 계층 생성
                               │
R2 주간 미러 (재해 복구 전용)   ─┘
frontmatter `raw_path`는 서버W 경로, `raw_r2`(선택)는 미러 URL
```

### 4.3 갱신 방식 — **초기 백필 + 일일 증분 + 주간 정합성**

각 옵션의 한계:

**"초기백필+일일증분"만**의 문제:
- 증분 크롤이 하루 실패하면 그 날 자료를 영영 놓칠 수 있음
- API 명세 변경·URL 패턴 변경 탐지 느림
- 삭제된 보도자료(드물지만 존재) 감지 불가

**"주기적 재스캔"만**의 문제:
- 재스캔 주기 동안 새 자료 노출 지연 (주간 재스캔 → 평균 3.5일 지연)
- 매번 전체 52,000건 훑는 건 크롤 예의 문제
- Git 커밋이 거대 배치로 쌓여 변경 추적 의미 퇴색

**권장 조합**:

| 빈도 | 작업 | 목적 |
|---|---|---|
| 1회 | 초기 백필 (5년치) | 기반 구축, 기간 10~14일 예상 |
| 매일 04:00 | 전일(T-1) 보도자료 증분 수집 | 신속 반영, 하루 ~30분 |
| 매주 월 06:00 | 최근 2개월 재스캔 | 증분 누락 복구 |
| 매월 1일 | 전체 manifest vs 스토리지 정합성 감사 | 유령 파일·누락 탐지 |

재스캔은 **최근 2개월로 제한**. 그 이전 자료는 변경이 거의 없고 (방금 검증 실험에서 측정 대기), 있어도 주간보다는 월간 재스캔에 맡긴다. 전체 5년 재스캔은 **장애 복구 시에만**.

## 5. 업데이트 파이프라인 상세

> **Govpress 재사용 원칙**: Govpress_md_pdf는 정책브리핑 **API 호출(크롤) + HWPX→MD 변환** 로직을 이미 코드로 보유. 웹서비스(api2.govpress.cloud)는 소량 on-demand 변환용이지 대량 백필용이 아니다. 따라서 Phase A는 Govpress **코드를 서버W 로컬에서 직접 실행**하고, 웹서비스는 건드리지 않는다.

```
                    ┌──────────────────────────────┐
                    │  korea.kr 정책브리핑 API     │
                    └──────────────┬───────────────┘
                                   │ (HTTPS, rate limit 준수)
                    ┌──────────────▼───────────────┐
                    │  Govpress crawler 코드 (로컬) │
                    │  - 날짜 범위, 전체 entity    │
                    │  - Govpress MCP가 bulk 모드로 실행   │
                    └──────────────┬───────────────┘
                                   │
              ┌────────────────────┼────────────────────┐
              ▼                    ▼                    ▼
     ┌────────────────┐  ┌──────────────────┐  ┌─────────────────┐
     │ HWPX → Govpress MCP 로컬│  │ Govpress hwpx→md │  │ manifest 갱신   │
     │ (data/raw)      │  │ 변환 코드 (로컬)  │  │                 │
     │ + sha256 기록   │  │ → data/md + fm   │  │                 │
     └────────┬────────┘  └────────┬─────────┘  └────────┬────────┘
              │                    │                      │
              └────────────────────┼──────────────────────┘
                                   ▼
                    ┌──────────────────────────────┐
                    │  Git commit & push           │
                    │  (메시지 규약: batch/correction/meta) │
                    └──────────────┬───────────────┘
                                   │
                    ┌──────────────▼───────────────┐
                    │  derive_hot.py (Govpress MCP 고유)   │
                    │  - briefings INSERT          │
                    │  - 청킹 + BGE-M3 임베딩      │
                    │    → Qdrant                  │
                    │  - statistics 추출 + Haiku   │
                    │    라벨링                    │
                    └──────────────────────────────┘
```

**Govpress 코드 재사용 방식** (실제 레포 조사 2026-04-17 확정):

Govpress는 두 레포로 쪼개져 있다. 재사용 경로가 각기 다르다.

| 레포 | 역할 | 패키지화 여부 | Phase A 편입 방식 |
|---|---|---|---|
| `gov-md-converter` | HWPX/PDF → MD 변환 엔진 | ✅ 패키지 `govpress-converter` (v0.1.11). `dist/govpress_converter-0.1.11-py3-none-any.whl` 사전 빌드 존재. 공개 레포는 인터페이스 컨트랙트만, 구현은 `_core/`에 비공개 | **editable install**: `pip install -e /home/USER/projects/gov-md-converter` (또는 wheel 설치) |
| `GovPress_PDF_MD` | korea.kr 정책브리핑 API 크롤러 + FastAPI 서버 + 캐시 | ❌ 설치 가능한 패키지 아님. `server/app/adapters/policy_briefing.py` 모듈이 서버 앱 내부에 박혀 있음 | **vendoring**: 해당 파일을 Govpress MCP `src/govpress_mcp/vendored/policy_briefing.py` 로 복사하거나, submodule 편입 후 `sys.path`에 서버 루트 추가 |

공개 API (실제 시그니처):

```python
# 변환 (gov-md-converter → govpress-converter 패키지)
import govpress_converter as gc
md_text: str = gc.convert_hwpx(path, table_mode="text")   # "text" | "html"
md_pdf:  str = gc.convert_pdf(path, timeout=300)          # Java 11+ 필요

# 크롤 (GovPress_PDF_MD/server/app/adapters/policy_briefing.py)
from policy_briefing import PolicyBriefingClient, PolicyBriefingItem
client = PolicyBriefingClient(service_key=os.environ["GOVPRESS_POLICY_BRIEFING_SERVICE_KEY"])
items: list[PolicyBriefingItem] = client.list_items(target_date)   # 하루치
file  = client.download_item_hwpx(item)                            # DownloadedPolicyBriefingFile
bytes_: bytes = file.content
```

중요 한계와 대응:

- `list_items()`는 **단일 날짜**만 받는다. 5년 백필은 Govpress MCP가 `for d in daterange(2021-04, 2026-04): client.list_items(d)` 루프로 돌린다.
- `PolicyBriefingCache.warm_item()`도 있지만 이건 Govpress 저장 디렉토리 구조에 묶여 있음. Govpress MCP는 이걸 쓰지 않고 **`list_items` + `download_item_hwpx`까지만 재사용**, 저장·변환·색인은 Govpress MCP가 직접.
- `api2.govpress.cloud` 웹서비스 HTTP 호출은 ✕ **절대 금지** (웹서비스는 소량 on-demand용, 대량 호출 시 자기 자신 과부하).
- `PolicyBriefingCache` 자체는 `{news_item_id}-{file_name}` 경로에 persistently 저장하긴 한다. 다만 실제 운영에서는 `prefetch_policy_briefings.py`가 **당일만** 돌리기 때문에 과거 데이터는 사실상 없음 → 5년 아카이브는 Govpress MCP가 새로 만들어야 한다는 결론은 변함 없음.

### 5.1 idempotency 원칙

같은 입력을 두 번 돌려도 결과가 같아야 한다. 크롤은 `(briefing_id, url)` 튜플을 unique key로. 다운로드 단계에서 기존 sha256과 비교해 동일하면 skip. Hot 계층 적재도 `INSERT OR REPLACE`.

### 5.2 크롤 rate limit

korea.kr API 호출 코드는 Govpress가 보유 → Govpress MCP가 로컬에서 bulk 모드로 실행. rate limit 정책은 Govpress 기본값을 따르되 bulk 실행 시 추가 가드를 건다.

- 동시성 5 이하
- 요청 간 0.3초 sleep
- User-Agent 명시 (Govpress 기본값 유지 또는 `govpress-mcp-bulk/1.0` 분리)
- 429·503 받으면 exponential backoff, 5분 후 재시도
- 크롤 상태(`last_seen_at`, 실패 큐, 재시도 카운터)는 Govpress MCP의 서버W 로컬 SQLite에 저장 (Govpress 웹서비스의 상태와 섞지 않음)
- bulk 실행 중 api2.govpress.cloud 웹서비스를 **절대 호출하지 않음** (자기 자신에 부하를 주면 안 됨)

건 수 감각:
- 전일 증분 (전체 entity): 수백~수천 건. 지자체 포함 시 하루 1,000건 초과할 수 있음 → 변환+임베딩 처리 시간 모니터링 필요.
- 5년 백필 (Phase A 상한 30만 건): **크롤 + 변환 + 임베딩 전부 처음부터**. API rate limit이 병목 → 실전 예상 2~4주. GPU 임베딩은 크롤과 병렬화하면 병목이 되지 않는다.

### 5.3 변환 단계 장애 대응 (HWPX→MD)

Govpress MCP는 Govpress 변환 함수를 import로 호출하므로, 실패 처리·재시도 로직은 Govpress MCP가 자체 구현.

- 변환 실패 시 frontmatter에 `extract_status: partial` 또는 `failed` 기록, Hot 계층 색인에서 제외
- Govpress MCP `retry_queue` 테이블에 기록 → 다음 날 재시도 (Govpress 코드가 개선됐을 수도 있음)
- 3회 연속 실패 시 수동 개입 알림 (Slack webhook 또는 GitHub 이슈 자동 생성)
- 실패 건도 원본 HWPX는 `data/raw/`에 보존 → Govpress 업데이트 후 일괄 재변환 가능
- Govpress 코드 개선은 **Govpress 레포에서 하고**, Govpress MCP는 submodule 커밋 SHA를 frontmatter `extracted_by`에 기록해 추후 선택적 재변환 가능

## 6. 재난 복구 시나리오

설계 검증은 **최악의 시나리오를 상정해 복구 경로를 확인하는 것**으로 한다.

| 시나리오 | 복구 경로 | 예상 소요 |
|---|---|---|
| Hot 계층 컨테이너 전체 손상 | `docker compose down && up` + `derive_hot.py` 재실행 (Warm 리포 기반) | 2~6시간 (GPU 임베딩 포함) |
| GitHub 리포(Warm) 유실 | 서버W Cold의 HWPX를 Govpress 엔진으로 전체 재변환 + Govpress MCP 재색인 | 3~7일 |
| 서버W 디스크 손실 (HWPX 포함) | 주간 R2/외장 미러에서 복원 → Govpress 재변환 → Govpress MCP 재색인 | 1~2일 (미러 복원 8~24h) |
| WSL 인스턴스 손상 (디스크는 정상) | `wsl --import`로 재설치 후 docker-compose 재기동, 데이터는 그대로 | 2~4시간 |
| Govpress 파이프라인 장애 | Govpress MCP는 기존 스냅샷으로 서빙 유지, 증분만 지연. 복구 후 catch-up 재스캔 | 장애 기간 + 재스캔 수 시간 |
| Qdrant 스토리지만 유실 | 임베딩 재계산 (BGE-M3 + RTX 3080, 30만 건 ≈ 1~2시간) | 1~2시간 |
| SQLite 손상 | frontmatter 기반 재빌드 | 30분~1시간 |
| Cloudflare Tunnel 장애 | 임시 `ngrok` 터널로 우회하거나 로컬 테스트 모드 | < 30분 |

가장 심각한 **서버W 디스크 + R2 백업 동시 유실**이 아니면 전부 복구 가능. R2 주간 미러가 단일 호스트 설계의 마지막 안전망.

OpenAI 임베딩으로 교체 시: 초기 백필 30만 건 × 평균 2.5K 토큰 × $0.02/1M ≈ **$15 일회성**, 이후 월 ~$3 추가. BGE-M3 셀프호스팅 대비 품질/운영 단순성 트레이드오프 잘 따져서.

서버W가 없었다면 fly.io/Render에 Qdrant+MCP 띄우는 데 월 $25~40 추가됐을 것. 자체 서버 활용이 가장 큰 비용 절감 요인.

## 8. 보안·공개 전략

**공개**: MD + frontmatter. 변환 스크립트. manifest. 스키마 정의.
**비공개**: statistics 테이블 (부처간 대조 로직이 드러나면 악의적 이용 가능성). embedding 원본. 내부 운영 로그.
**원본 HWPX**: 서버W 로컬에 두되 Docker 컨테이너엔 ro 마운트로만 노출. 제3자 요청 시 "정책브리핑 공식 채널을 이용하세요"로 안내. R2 백업 미러도 인증된 접근만.

저작권: 보도자료 본문은 「공공누리 1유형(출처표시)」이 대부분. frontmatter에 원본 URL 명시하면 안전. 다만 로고·인물사진은 변환 시 제외.

## 11. Govpress 베이스 프로젝트와의 관계

> 핵심 정정: Govpress_md_pdf는 크롤·변환 **로직을 코드로 보유**하되, 웹서비스(api2.govpress.cloud)는 소량 on-demand용이며 저장 기능이 없다. 따라서 대량 백필은 Govpress 코드를 Govpress MCP 프로젝트에 편입해 **로컬 bulk 모드**로 실행하고, 저장·색인은 Govpress MCP가 자체 구축.

### 11.1 역할 구분

| 크롤 (korea.kr API 호출 로직) | Govpress | Govpress MCP bulk_ingest.py (로컬) | Govpress MCP cron (로컬) |
| HWPX → MD 변환 로직 | Govpress | Govpress MCP bulk_ingest.py (로컬) | Govpress MCP cron (로컬) |
| HWPX 원본 보관 | Govpress MCP | — | — |
| MD + frontmatter 보관 | Govpress MCP | — | — |
| SQLite + FTS5 인덱스 | Govpress MCP | — | — |
| 벡터 임베딩 + Qdrant | Govpress MCP | — | — |

### 11.3 충돌 방지 원칙

- **단방향 의존**: Govpress MCP → Govpress는 크롤·변환 함수만 호출. Govpress 웹서비스는 Govpress MCP 존재를 몰라도 동작.
- **웹서비스 비호출**: Govpress MCP bulk·cron은 **api2.govpress.cloud를 절대 호출하지 않음**. 같은 Govpress 코드를 로컬에서 import로만 사용. 네트워크 로그로 회귀 검증.
- **리소스 격리**: Govpress MCP Docker 네트워크는 별도(`govpress_mcp_net`). GPU 사용 시간대 조율, Govpress 웹서비스가 GPU를 쓰면 bulk 실행 창을 야간으로.
- **Govpress 코드 버전 고정**: submodule SHA 또는 pip editable install의 커밋 해시를 `extracted_by` frontmatter 필드에 기록. Govpress 업데이트 시 선택적 재변환 가능.
- **배포 독립**: Govpress MCP 재시작이 Govpress 웹서비스에 영향 없어야 (별도 컨테이너, 별도 cloudflared route).
- **로그 분리**: api2.govpress.cloud 로그와 mcp.govpress.cloud 로그, bulk_ingest 로그는 다른 파일에.

### 11.4 Phase A 영향

Govpress가 크롤·변환 코드를 보유하고 있어 Govpress MCP는 **같은 로직을 처음부터 구현할 필요가 없다**. 다만 Govpress 웹서비스에는 자료가 없으므로 실행·저장·색인은 Govpress MCP가 전부 수행.

Phase A 시퀀스:

1. Govpress 레포 submodule 편입 + 함수 시그니처 확인 (Day 2)
2. `bulk_ingest.py`로 korea.kr API를 통해 5년치 전체(중앙+광역) 메타 조회
3. HWPX 다운로드 → `data/raw/`에 보관
4. Govpress 변환 함수로 MD 생성 → `data/md/`에 보관 + Git 커밋
5. derive_hot.py로 SQLite·Qdrant 색인
6. `policy-briefing-kr` 공개 리포에 MD 푸시

예상 소요: **2~4주** (API rate limit이 병목. GPU 임베딩·변환은 병렬로 돌려도 여유).

Govpress 코드 이득: 크롤 엔드포인트·인증·파싱·HWPX 구조 처리 등은 이미 Govpress에서 검증된 구현을 쓰므로 실패 리스크가 낮다. Govpress MCP는 orchestration(배치 큐, 재시도 기록, 저장 경로 규약)만 추가.
