# GeekNews Show GN 제출 초안

GeekNews Show GN의 히트 패턴은 (1) 한 줄 훅, (2) 내가 왜 만들었나, (3) 직접 보여주는 GIF·링크, (4) 짧은 기술 스택, (5) 반응 유도 질문. 아래 본문은 이 구조를 따른다.

---

## 제목 (3안 중 택 1)

**A안 (재작성 · 추천)**
> Show GN: Govpress MCP — 중앙부처 보도자료 5년치를 Claude가 시계열로 읽게 하는 MCP (정책의 git log + 부처간 diff)

**B안 (기능 강조)**
> Show GN: 부처간 엇박자를 기계가 찾아주는 MCP — 보도자료 5년치 분석 서버

**C안 (문제 제기 훅)**
> Show GN: "AI 인재 50만 vs 20만" — 부처별 수치 충돌 자동 적발 도구를 만들었습니다

→ 추천은 **A안**. 초기안의 "git blame"은 정정 보도자료 빈도가 낮다는 도메인 사실에 부합하지 않아 "git log + 부처간 diff"로 교체. 두 해시태그가 각각 `trace_policy`·`cross_check_ministries`에 1:1 대응해 오버셀링을 피한다.

## 본문 (약 450자)

> 정부 보도자료는 매일 100건 넘게 쏟아지지만, 5년치를 한 번에 읽는 사람은 아무도 없습니다. 그래서 같은 정책의 숫자가 부처별로 엇나가고(과기정통부 "AI 인재 50만", 교육부 "20만"), 목표 연도가 조용히 1년씩 밀려도 아무도 짚지 않습니다.
>
> **Govpress MCP**는 5년치 중앙부처 보도자료를 HWPX→Markdown으로 변환해 Git에 공개하고, Claude가 이 아카이브를 맥락으로 쓰도록 7개 도구를 MCP로 노출합니다. 핵심은 3개:
>
> - `trace_policy` — "디지털플랫폼정부 어떻게 변했어?" → 5년 시계열 자동 요약
> - `cross_check_ministries` — "AI 인재양성 부처간 엇박자 있나?" → 수치 충돌 후보 적시
> - `validate_statistics` — 내일 배포할 보도자료 초안을 과거 24개월 발표와 대조
>
> 정책연구자·기자·공공기관 실무자가 각각 다른 도구를 쓰게 설계했고, 특히 세 번째는 오보 리스크를 배포 **전에** 막는 B2G 훅입니다.
>
> 스택: Python `mcp` SDK + SQLite FTS5 + Qdrant(BGE-M3) + Redis. Claude Desktop에서 `uvx govpress-mcp` 한 줄로 설치됩니다.
>
> 리포: https://github.com/USER/govpress-mcp
> 데이터: https://github.com/USER/policy-briefing-kr (공개)
>
> 피드백, 특히 "이 도구는 이래야 한다"는 의견 환영합니다.

**문자 수: 한글 기준 약 470자, 본문만 2~3분 안에 스캔됨 — Show GN 평균 반응 구간**

## 댓글 1번 (자문/자답 — 작성자 선점)

첫 댓글을 작성자가 먼저 다는 것이 Show GN 관례. 아래 3종 중 분위기에 맞게 하나 선택.

**버전 1 — 기술 FAQ 선점**

> 자문자답 몇 개:
>
> **Q. 왜 RAG 말고 MCP?**
> A. Claude가 "언제 어떤 데이터를 볼지" 스스로 정하게 하고 싶었습니다. 단순 RAG로 뭉쳐 넣으면 5년치를 한 컨텍스트에 압축하면서 시계열 구조가 뭉개집니다. 도구로 쪼개두면 Claude가 trace_policy→get_briefing→cross_check로 단계적 탐색을 합니다.
>
> **Q. 정책브리핑 API 있잖아?**
> A. 그건 "오늘 뭐 나왔어" 용도고, Govpress MCP는 "5년간 어떻게 변했어" 용도입니다. 다른 축입니다.
>
> **Q. 중앙부처만?**
> A. 지자체·공공기관은 로드맵에. 데이터량이 3배 되므로 인덱스 설계부터 다시 해야 해서 초기 범위에서 뺐습니다.

**버전 2 — 동기/서사 선점**

> 만든 계기: 작년에 기자 친구가 "AI 인재 발표가 부처마다 다른데 어느 쪽이 맞는지 확인하려면 3일 걸린다"고 하더군요. 그 3일을 30초로 줄이는 게 목표였습니다. validate_statistics는 원래 없던 기능이었는데, 중앙부처 전직자 분이 "배포 전에 이걸로 체크하면 고과가 바뀐다"고 해서 우선순위를 올렸습니다.

**버전 3 — 기여 유도 선점**

> `good first issue`로 몇 개 남겨두었습니다:
> - 부처명 alias 보강 (특히 시기별 개편: 산자부→산업부→산업통상자원부)
> - 새 숫자 단위 패턴 추가 (GWh, m³, 톤·CO₂eq 등 환경/에너지 분야)
> - 도구별 벤치마크 질의 세트 확장 (현재 30개 → 100개 목표)
>
> 벡터 DB·한국어 NLP·정책연구 관심 있으시면 편하게 이슈 남겨주세요.
